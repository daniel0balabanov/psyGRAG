# Retrieval Pipeline: технические детали

Описание четырёх ключевых компонентов RAG-пайплайна системы.

---

## 1. BGE-M3 — эмбеддинги

**Модель:** `BAAI/bge-m3`  
**Код:** `app/indexing/embeddings.py`

BGE-M3 — multilingual dense embedding model от BAAI. Поддерживает 100+ языков, включая русский.

### Что делает

Переводит текст в вектор фиксированной размерности (1024-dim), при этом семантически похожие тексты получают близкие векторы.

### Как используется в пайплайне

- **При индексации** (`IndexPipeline`): каждый чанк документа превращается в вектор и сохраняется в ChromaDB.
- **При запросе** (`QueryPipeline`): пользовательский вопрос также векторизуется (`embed_query`), затем ищутся ближайшие векторы.

### Ключевые параметры

| Параметр | Env | Дефолт |
|---|---|---|
| Модель | `PSY_EMBEDDING_MODEL_NAME` | `BAAI/bge-m3` |
| Устройство | `PSY_EMBEDDING_DEVICE` | `cuda:0` |
| Batch size (при индексации) | `PSY_EMBEDDING_BATCH_SIZE` | `16` |

### Нормализация

Векторы нормализуются (`normalize_embeddings=True`) — это позволяет использовать косинусное сходство через обычное скалярное произведение, что ускоряет поиск.

## 2. ChromaDB — dense retrieval

**Библиотека:** `chromadb` (PersistentClient)  
**Код:** `app/indexing/chroma_store.py`  
**Данные:** `data/chroma/`

ChromaDB — векторная база данных. Хранит эмбеддинги чанков и выполняет приближённый поиск ближайших соседей (ANN).

### Что делает

Принимает вектор запроса → возвращает `top_k` чанков с наименьшим расстоянием (косинусным).

### Как используется

```
query_embedding → chroma_store.query(embedding, top_k=18) → [RetrievalHit, ...]
```

Результаты содержат `score = 1.0 - distance` (чем выше — тем релевантнее).

### Структура хранимых данных

Каждый чанк хранится как:
- `id` — уникальный `chunk_id`
- `document` — текст чанка
- `embedding` — вектор (BAAI/bge-m3)
- `metadata` — `doc_id`, `source`, `title`, формат, и другие поля

### Параметры профиля

| Профиль | `dense_top_k` |
|---|---|
| fast | 10 |
| balanced | 18 |
| quality | 26 |

## 3. BM25 — sparse retrieval

**Библиотека:** `rank_bm25` (BM25Okapi)  
**Код:** `app/indexing/bm25_store.py`  
**Данные:** `data/bm25/index.pkl`

BM25 (Best Match 25) — классический алгоритм ранжирования на основе TF-IDF с нормализацией длины документа.

### Что делает

Ищет чанки по точному совпадению токенов (ключевых слов). Хорошо работает там, где dense retrieval даёт сбои — например, на специфических терминах, именах, аббревиатурах, диагнозах.

### Токенизация

```python
TOKEN_RE = re.compile(r"\w+", re.UNICODE)
tokens = [token.lower() for token in TOKEN_RE.findall(text)]
```

Простая unicode-токенизация без стемминга. Подходит для многоязычных текстов (русский/английский).

### Как используется

```
query → _tokenize(query) → bm25.get_scores() → top_k чанков
```

Индекс строится один раз при индексации и сохраняется через `pickle`. При запросе загружается в память (`_ensure_bm25`).

### Параметры профиля

| Профиль | `sparse_top_k` |
|---|---|
| fast | 10 |
| balanced | 18 |
| quality | 24 |

### Ограничения

- Не учитывает семантику (только точные слова)
- При очень коротких запросах (1-2 слова) может давать нерелевантные результаты
- `pickle` — не потокобезопасен при параллельных записях

## 4. RRF + Cross-Encoder Reranking

### 4.1 Reciprocal Rank Fusion (RRF)

**Код:** `app/retrieval/hybrid.py`

RRF — алгоритм слияния нескольких ranked lists в один без нормализации исходных скоров.

#### Формула

```
score(doc) = Σ  1 / (k + rank_i(doc))
```

где `k = 60` (константа, сглаживающая влияние топовых позиций), `rank_i` — позиция документа в i-м списке.

#### Почему RRF, а не простое усреднение скоров

- Dense и sparse скоры имеют разные шкалы (косинусное сходство vs BM25)
- RRF работает только с позициями — не нужна нормализация
- Хорошо проверен в литературе по information retrieval

#### Пример слияния

```
Dense:  [A(1), B(2), C(3), D(4)]
Sparse: [C(1), A(2), E(3), B(4)]

RRF score A = 1/61 + 1/62 = 0.0327
RRF score C = 1/63 + 1/61 = 0.0322
RRF score B = 1/62 + 1/64 = 0.0317
```

Итог: A → C → B (объединяет оба сигнала).

---

### 4.2 Cross-Encoder Reranking

**Модель:** `BAAI/bge-reranker-v2-m3`  
**Код:** `app/retrieval/rerank.py`

Cross-encoder — модель, которая оценивает пару `(query, document)` совместно, в отличие от bi-encoder (BGE-M3), который кодирует их отдельно.

#### Почему cross-encoder точнее

| | Bi-encoder (BGE-M3) | Cross-encoder (reranker) |
|---|---|---|
| Скорость | Быстро (векторы вычислены заранее) | Медленно (каждая пара отдельно) |
| Точность | Средняя | Высокая |
| Применение | Поиск по всему корпусу | Переранжирование топ-N кандидатов |

#### Как используется

```
fused_hits[:rerank_top_n] → reranker.rerank(query, hits, top_n) → [RetrievalHit, ...]
```

Reranker получает `rerank_top_n` кандидатов после RRF и возвращает `rerank_top_n` переупорядоченных по релевантности.

#### Параметры профиля

| Профиль | `rerank_top_n` | `final_context_chunks` |
|---|---|---|
| fast | 8 | 4 |
| balanced | 14 | 6 |
| quality | 20 | 8 |

## Полная схема пайплайна

```
Запрос пользователя
       │
       ▼
  BGE-M3 embed_query
       │
  ┌────┴────┐
  │         │
  ▼         ▼
ChromaDB   BM25
(dense)   (sparse)
top_k      top_k
  │         │
  └────┬────┘
       │
       ▼
     RRF fusion
  (объединение списков)
       │
       ▼
  Cross-Encoder
   reranking
       │
       ▼
  top-N чанков
  → промпт → Ollama (Gemma)
       │
       ▼
  Ответ + цитаты
```
