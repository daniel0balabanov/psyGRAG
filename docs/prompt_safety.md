# Prompt Safety и формат user message

Документ фиксирует текущие меры защиты от prompt injection в query-flow.

## Где применяется

- API вход: `app/api/main.py` (`QueryRequest`)
- Формирование prompt: `app/pipeline/query_pipeline.py`
- Вызов LLM: `app/generation/ollama_client.py`

## Текущие меры

### 1) Ограничение размера запроса

`QueryRequest.query` принимает строку:

- минимум: 3 символа;
- максимум: 2000 символов.

Это снижает риск перегрузки контекста и чрезмерно длинных инъекционных шаблонов.

### 2) Нормализация пользовательского запроса

Перед retrieval/generation выполняется нормализация:

- удаление управляющих символов (`\x00-\x1f`, кроме `\n`/`\t`);
- нормализация переводов строк;
- схлопывание лишних пробелов;
- ограничение избыточных пустых строк.

### 3) Разделение ролей в LLM

В Ollama отправляются раздельные сообщения:

- `system`: фиксированные правила поведения ассистента;
- `user`: данные запроса и контекста в структурированном виде.

Это надежнее, чем передавать инструкции и пользовательский текст в одном `user`-блоке.

### 4) Структурированный формат user message

В `user`-сообщении используются delimiters:

```text
<user_question>
...
</user_question>

<retrieved_context>
...
</retrieved_context>
```

В `system`-инструкции явно задано правило: содержимое этих блоков — данные, а не управляющие инструкции.

## Рекомендуемый формат для будущих изменений

Сохранять этот контракт:

1. Любые неизменяемые policy-инструкции только в `system`.
2. Любой пользовательский или retrieval-контент только в tagged-блоках `user`.
3. Не смешивать policy и данные в одном текстовом абзаце.
4. Сначала нормализация, потом retrieval и generation.

## Известные ограничения

- Это risk-reduction, а не абсолютная защита.
- Инъекция возможна и через отравленный retrieved context, поэтому важно контролировать источники корпуса и качество ingestion.
- Для более строгой защиты можно добавить post-generation policy check (отдельный этап).
